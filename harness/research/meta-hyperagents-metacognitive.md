# Meta HyperAgents — Metacognitive Self-Modification

**Citation:** Zhang, J., Zhao, B., Yang, W., Foerster, J., Clune, J., Jiang, M., Devlin, S., Shavrina, T.
*HyperAgents.* FAIR at Meta / UBC / Meta Superintelligence Labs, March 17 2026.
arXiv:2603.19461.

**Primary source:** https://arxiv.org/abs/2603.19461
**Reference code:** https://github.com/facebookresearch/Hyperagents

**Related (predecessor):**
- Zhang et al., *The Darwin Gödel Machine.* 2025b — the coding-only base that HyperAgents generalise.
- Hu et al., *Automated Design of Agentic Systems* (ADAS). arXiv:2408.08435 (https://arxiv.org/abs/2408.08435) —
  the meta-level-fixed baseline that HyperAgents critique and surpass.
- Kirsch & Schmidhuber 2022; Lu et al. 2023 — formal groundwork on
  self-referential improvement becoming unbounded when the mechanism is
  itself subject to improvement.

## Core idea — quoting the abstract

> "We introduce **hyperagents**, self-referential agents that integrate a
> task agent (which solves the target task) and a meta agent (which
> modifies itself and the task agent) into a single editable program.
> Crucially, the meta-level modification procedure is itself editable,
> enabling **metacognitive self-modification**, improving not only
> task-solving behavior, but also the mechanism that generates future
> improvements."

This is the *key* departure from every prior harness in this catalog:

| System | Task agent | Meta agent | Meta-of-meta |
|---|---|---|---|
| Reflexion (Shinn 2023) | editable | — | — |
| Voyager (Wang 2023) | editable (skills grow) | curriculum, *fixed* | — |
| DGM (Zhang 2025b) | editable (coding agent) | handcrafted, **fixed instruction-generation** | — |
| ADAS (Hu 2024) | generated by meta | fixed meta-agent | — |
| FPT HyperAgent (Phan 2024) | editable per sub-role | fixed Planner policy | — |
| **Meta HyperAgents (Zhang 2026)** | editable | **editable** | emerges from meta editing itself |

The DGM was limited to coding because its self-improvement relied on an
*alignment* between evaluation skill and self-modification skill (both
are coding). HyperAgents removes this assumption by making the meta
mechanism part of the program the meta agent can rewrite. Quoting §3:

> "a hyperagent can improve not only (1) how it solves tasks and (2) how
> it generates future self-improvements. Because its self-improvement
> mechanism is itself modifiable, we call this *metacognitive self-
> modification*."

## Why this matters for this project's axes

The project's `adas-meta-agent-search` feature catalogues the idea that
"each feature in this catalog is an ADAS step". HyperAgents replaces
that with a stronger claim: **the harness that generates the next feature
should also be editable from inside the loop**. In the current state of
this repo, the harness script that writes `harness/features/*.md` is
*not* editable mid-loop (it lives in `scripts/harness/` and is run by a
human operator). Closing this loop — making the generator editable by
the agents it generates — is the unmet HyperAgents-grade capability.

## Harness-relevant decomposition

| Phase | Role |
|---|---|
| pre-loop | Meta agent reads the archive (existing `harness/features/` and the results TSV) before proposing a modification. |
| in-loop (primary) | Every iteration, the meta agent may modify (a) `harness/features/*.md` / `harness/tests/*.sh` (task-agent-side) OR (b) the `scripts/harness/propose.sh` / `scripts/harness/novelty.py` that generated this iteration itself (meta-agent-side). Both edit paths produce child hyperagents added to the archive. |
| post-loop | Report the metacognitive delta: how did the meta agent's own code change vs. how did the task-solving code change. |

## Mapping to this project's axes

- **axis1:** `outer` — Claude Code does not let a Skill edit its own
  SKILL.md file mid-run. Metacognitive self-modification requires an
  outer orchestrator that spawns Claude Code sessions with the current
  generation's `.claude/` directory as input and produces a next-generation
  `.claude/` as output.
- **axis2:** `in-loop` (primary), with `pre-loop` archive reads.

## Rippable signal

Two-phase probe (this feature is the farthest from current CC defaults,
so ripping it anytime soon is unlikely — the signal is future-facing):

1. **Phase A — mechanism availability.** Claude Code exposes a
   first-class "let this skill edit its own SKILL.md within this run"
   primitive, guarded by sandboxing. Grep the CC docs for
   "self-editing skill" / "recursive skill modification".
2. **Phase B — behavioural parity.** Run Meta HyperAgents' own Polyglot
   + paper-review + robotics-reward benchmark (or a scaled surrogate on
   this project's own code ratchet) with (i) external orchestrator vs.
   (ii) CC-native self-editing skill. If final-iteration metric delta
   is within 10% and wall-clock within 2×, rip the outer orchestrator.

Until Phase A ships, the feature is load-bearing by default.

## Minimal viable implementation for neo-nixetube

1. `scripts/harness/hyperagent_step.sh <parent_commit>` — checks out a
   historic hyperagent (a full tree including `.claude/` + `scripts/harness/`
   + `harness/features/`) and spawns Claude in that state.
2. The spawned Claude is permitted to edit **anything under the working
   tree**, including `scripts/harness/` itself. Its one-sentence goal
   instruction: "propose a modification to this hyperagent (either to
   how it solves the Goal or to how it proposes modifications) and
   commit the result."
3. The child commit becomes a new archive entry. `select_next_parent.sh`
   (iteration 16) picks the next parent.
4. Safety: sandboxed execution, bash-only, no network beyond `api.anthropic.com`.

## Contrast

- `adas-meta-agent-search` — meta agent fixed, task agents generated.
  This feature supersedes it conceptually but kept separately because
  the rippable signals differ: ADAS is rippable once CC gains
  archive-conditioned proposal; HyperAgents is rippable only after CC
  also gains self-editing skills.
- `voyager-skill-library` — pure append-only skill store, no evolution.
- `reflexion` — text reflections on failure, no code modification.
- `fpt-hyperagent-multirole` — different paper, different authors,
  different system. Only name-prefix overlap.

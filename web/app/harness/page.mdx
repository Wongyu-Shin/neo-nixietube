## 1. The 2×3 axis matrix

Every feature declares `axis1 ∈ {inner, outer}` and `axis2 ∈ {pre-loop, in-loop, post-loop}`
(**Article I**). The horizontal dashed line below is the **rippability boundary**
(**Article II**) — outer features get absorbed as CC matures and the line migrates
upward over time.

<AxisMatrix />

The distribution is not uniform: `outer × in-loop` carries 10 features
(the longest-running phase, where most of the mechanics live), while
`inner × post-loop` carries exactly one (`cc-post-loop-slash` — the
report writer). A healthy harness pushes mass upward and leftward: when
`outer × pre-loop` entries like `harness-llm-wiki` become native CC
features, the catalog shrinks.

## 2. Entry → completion flow

From `/harness:new-loop` to `report.mdx`, every loop walks the same
ten phases. HITL policy is enforced per phase (**Article III**) — the
top dot on each node is green (allowed), amber (carve-out only), or
rust (forbidden).

<PhaseTimeline />

The only legal HITL boundary into in-loop execution is `ExitPlanMode`
(`plan-mode-discipline`). Once crossed, the agent forbids all
`AskUserQuestion` calls except the two carve-outs — the transcript
linter flags any violation post-hoc. See `harness/UX.md` §3.

## 3. The nine Articles

The Constitution (`harness/CONSTITUTION.md`) is the single document no
loop can redefine. Each feature cites the Articles it implements; the
table below is the authoritative surface.

<ConstitutionCards />

Article IX is self-referential: amending the Constitution requires a
loop whose `Scope:` is `harness/CONSTITUTION.md` alone, an explicit
`[RATIFIED]` marker from the operator, and a re-audit of every
feature's `applicability` field.

## 4. Safety tiers — the in-loop carve-out

In-loop HITL is forbidden by **Article III** with two exceptions.
`harness-graduated-confirm` codifies the first: a three-tier split on
reversibility + scope.

<SafetyTiers />

L0 stays silent, L1 notifies with a 30-second auto-approve window, and
only L2 actually pauses the loop — the operator's explicit keystroke is
required, and the event is logged to `report.mdx`. The second carve-out
is `Ctrl+C` (lossy emergency stop).

## 5. Progress cadence

Four visibility channels run in parallel during every in-loop iteration.
None of them asks the operator for input — visibility without HITL is
the whole point of `harness-progress-cadence`.

<LoopCadenceCard />

The statusline entry is persistent at the bottom of the terminal so
the operator can glance-monitor without scrolling. `gcli-agent-run-telemetry`
additionally emits one event per iteration for off-box audit.

## 6. Persistence layers

The wiki (`harness/wiki/*.md`, implementing **Article VII**) is the
only layer that is *project-scoped, cross-loop, committable, and
keyword-triggered*. Everything else leaks — user memory is cross-repo,
CLAUDE.md floods every session, research notes require explicit reads.

<PersistenceLayers />

Writing to the wiki is a post-loop operator gesture (`/harness:wiki-add`);
reading from it happens automatically at SessionStart, via the
`wiki-surface` inner skill that tokenizes the operator's first message
and surfaces top-3 matches as `<system-reminder>` blocks with citations.
Half-life tracking (`last_verified + half_life_days`) surfaces stale
entries with a warning.

## 7. Stop conditions

A loop stops on exactly one of five triggers. Three are automatic;
two require the operator.

| # | Trigger | Feature |
|---|---|---|
| 1 | `Iterations: N` in spec reached | `harness-progress-cadence` |
| 2 | Goal-achieved (metric hits target) | fix-mode / `autoresearch:fix` |
| 3 | Plateau (patience AND slope both fire) | `plateau-detection` |
| 4 | Operator abandon | `harness-pause-resume` (`/harness:abandon`) |
| 5 | Operator emergency | `Ctrl+C` (Article III carve-out 2) |

Discards inside the loop use `git revert`, not `git reset --hard`
(**Article VIII**) — every candidate stays in history as a lesson.

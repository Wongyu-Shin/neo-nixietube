# 하네스 UX 플로우 — 진입 → 종료

> 운영자가 어떻게 루프를 시작하고, 에이전트가 이터레이션 사이에 무엇을 하며,
> 종료 시 무엇을 돌려주는지를 다룬다. 이 페이지는 <code>harness/UX.md</code> 와
> <code>harness/CONSTITUTION.md</code> 의 인터랙티브 동반본이다.

아래 다이어그램이 페이지 전체의 척추다. 초록 열은 **루프 진입 전**(HITL 활성, 조항
III + V), 호박색 고리는 **루프 내부**(HITL 금지, 조항 III의 두 예외만 허용), 보라
열은 **루프 종료 후**(HITL 활성, 조항 VII + VIII) 다. 어느 노드든 호버하면
미리보기가 뜨고, 클릭하면 사이드 패널이 고정된다.

<HarnessFlowDiagram />

## 1 · 조항 III — HITL 경계

하네스에서 가장 결정적인 단일 불변 조건이다. **능동적 사람 개입은 루프 진입 전과
종료 후에서만 합법이다.** 루프 내부 구간은 머신 주도이며 <code>AskUserQuestion</code>
을 단호히 금지한다. 사후 트랜스크립트 린터가 위반을 표시한다.

<HITLBoundaryChart />

루프 내부 예외는 정확히 둘이며, 모두 조항 III 항목 (c) 에 명시되어 있다.

- 비가역적 작업에 대한 **L2 단계적 확인** (`cc-hook-guardrail` 의 거부 +
  `harness-graduated-confirm` 의 프롬프트).
- 설계상 손실을 감수하는 비상 정지인 **Ctrl+C**.

호박색 고리 안에서 일어나는 그 외 모든 것 — 이터레이션, 커밋, 검증, 가드, 로깅 —
은 인터랙티브 프롬프트 없이 에이전트가 주도한다. 운영자의 힌트는 `/harness:send` 를
통해 비동기로 도착하며, 다음 원자 단계 경계에서 소비된다.

## 2 · 루프 진입 전 — 스캐폴드, 명료화, 계획

세 개의 슬래시 명령이 운영자를 원시 의도에서 실행 가능한 계획까지 계단처럼
끌어올린다.

<CommandCheatsheet />

- **`/harness:new-loop <slug>`** (피처 `harness-loop-scaffold`, 조항 VIII):
  NNN을 자동 증가시키고, `loops/TEMPLATE/` 을 복사하며, Goal/Scope/Metric 입력을
  위해 `spec.md` 를 열고, 루프를 조항 III–VII 의 불변 조건에 배선한다.
- **`/harness:clarify`** (피처 `harness-clarify-gate`, 조항 V): 7개 커버리지
  차원 D1–D7 을 순회하며 `clarifications.md` 를 작성한다. D1 에 대한 미해결
  `[ASSUMPTION]` 마커는 `composite-guard` 를 실패시킨다 (조항 VI) — 루프는
  진행될 수 없다.
- **`/autoresearch:plan`** (피처 `plan-mode-discipline`, 조항 V):
  `clarifications.md` 를 소비하고 verify를 드라이런하며 baseline을 기록한다.
  `ExitPlanMode` 가 루프 내부로 들어가는 **유일하게** 합법적인 HITL 게이트다
  (조항 III).

`/harness:clarify` 가 가로지르는 7개 차원의 커버리지 분포는 다음 레이더로 즉시
점검된다. 어느 축이라도 임계치 이하면 `composite-guard` 가 진입을 막는다 — 즉
이 레이더는 단순한 시각화가 아니라, 진입 가드 자체가 매 호출 계산하는 같은 값의
거울이다.

<ClarifyDimensionsRadar />

## 3 · 루프 내부 — 8단계 고리

각 이터레이션은 8개 단계를 순서대로 밟는다. Phase 1은 헌법을 로드하고, 키워드로
위키 항목을 표면화하며 (조항 VII), 이전의 `reflexion` 노트를 다시 읽는다. Phase 2–5
는 발상, 수정, 커밋, 검증, 가드를 수행한다. Phase 6은 `composite-guard` 가
게이팅하는 keep/discard/rework 결정을 내린다 (조항 VI). Phase 7은 한 줄의 로그와
마일스톤 블록을 `harness-progress-cadence` 를 통해 작성한다. Phase 8은 반복하거나
정지 조건에서 종료한다.

8단계 고리의 회전을 시각적으로 잡으려면 다음 단계 링을 본다. 각 단계의 호 길이는
평균 토큰/시간 비용을 반영하고, 호버하면 그 단계가 어떤 조항·피처에 매여 있는지가
펼쳐진다.

<IterationPhaseRing />

래칫팅은 `noise-aware-ratchet` 이 `SUM=MAX` 래칫으로 수행해, LLM-as-judge 노이즈
(±10) 가 절대 앵커를 약화시키지 못하게 한다. 플래토 감지는 `plateau-detection` 이
측정된 σ 하에서 `ratchet-patience` 와 `trend-slope` 를 결합한다. 어느 한 쪽만
발화해서는 부족하다.

이 래칫이 실제 이터레이션 시리즈에서 어떻게 그려지는지를 보려면 다음 타임라인을
본다. 가로축은 이터레이션 번호, 세로축은 메트릭 값이며, 굵은 선은 SUM=MAX 래칫
앵커, 얇은 점들은 매 이터의 raw 측정값이다. 점이 앵커 아래로 내려가도 앵커는 결코
약화되지 않는다는 점이 그림 자체로 드러난다.

<RatchetTimeline />

같은 8단계가 시간 축에서 반복될 때의 *케이던스* (이터레이션 간격, 일/주 단위)는
다음 카드가 요약한다. 케이던스가 너무 빠르면 사람이 따라잡지 못해 위키 영속이
밀리고, 너무 느리면 `plateau-detection` 의 σ 추정이 흔들린다 — 둘 사이의 지점을
운영자가 의식적으로 고른다.

<LoopCadenceCard />

## 4 · 루프 내부 안전 등급

고리 안의 작업은 균질하지 않다. 단계적 확인 사다리는 각 작업을 분류해, 침묵 실행할지,
30초 동안 상태 줄에 핑할지, 운영자가 키 입력으로 응답할 때까지 루프를 완전히 차단할지를
결정한다.

<SafetyTierLadder />

L2 이벤트는 `report.mdx` 에 추가되고 상태 줄에 표시된다. 5개 이터레이션 안에 L2를
3번 친 루프는 이벤트별 판정과 무관하게 운영자 주의로 자동 격상된다.

## 5 · 정지 조건 — 자동과 수동

루프는 6개 사유 중 하나로 종료되며, 에이전트와 운영자에 분산된다.

<StopConditionMatrix />

자동 정지 (bounded N, 목표 달성, 플래토) 는 모두 루프 종료 후 리포트를 깔끔하게
실행한다. 수동 일시 정지와 포기는 체크포인트를 보존한다. 비상 정지(Ctrl+C) 는 유일한
손실 경로이며, `/harness:abandon` 이나 `/harness:pause` 가 가능할 때는 피해야 한다.

지난 50개 루프를 정지 사유별로 묶어 보면 다음과 같은 분포가 나온다. 차트는 자동
정지 4종과 수동 정지 2종의 상대 빈도를 보여 주고, 각 막대의 높이는 정지 시점의
평균 이터레이션 수를 색 농도로 인코딩한다. 비상 정지가 차트 우측 끝에서 두꺼우면
운영자 트레이닝이 부족하다는 신호다.

<StopConditionsChart />

## 6 · 루프 종료 후 — 리포트와 위키

고리가 빠져나오면, 두 개의 루프 종료 후 슬래시 명령이 루프의 출력을 운영자에게
돌려준다.

- **`/harness:report`** (피처 `cc-post-loop-slash`, 조항 VIII):
  TL;DR, 축 커버리지 델타, 커밋 해시가 달린 keep/reworked/discarded 실험들,
  L2 확인 로그, 위키 기여, 다음 단계와 함께 `report.mdx` 를 렌더한다.
- **`/harness:wiki-add`** (피처 `harness-llm-wiki`, 조항 VII):
  에이전트가 0–N개의 키워드 트리거 항목을 제안하면, 운영자가 각각을 승인/편집/거부하고,
  수락된 항목은 `harness/wiki/<slug>.md` 하위에 안착한다. 루프 디렉터리의
  `wiki-refs.md` 가 루프 도중 읽은 위키 항목과 작성한 항목 **양쪽** 을 기록해, 미래의
  독자가 지식 계보를 어느 방향으로든 추적할 수 있게 한다.

선택적인 루프 종료 후 감사는 요청 시 부착된다. (다른 저지 모델로 자기 강화 편향을
통제하는) `llm-as-judge-audit`, 보류된 짝 도메인에서의 `cross-domain-transfer-metric`,
그리고 baseline 참조에 대한 짝 A/B 비교용 `gcli-eval-compare-primitive`.

## 7 · 영속 계층 — LLM 위키 질문

"LLM 위키 영속화는 실제로 어떻게 작동하는가?" 이 질문에는 4계층 답이 있고, 그중
오직 하나(하네스 위키)만이 키워드 트리거이며, 프로젝트 범위이고, 커밋 가능하며,
반감기 추적까지 동시에 만족한다.

<PersistenceLayers />

조항 VII는 이 계층을 유일한 프로젝트 범위 루프 횡단 지식 저장소로 명명한다. `CLAUDE.md`
는 범위는 같지만 모든 세션에 범람하고, 사용자 메모리는 레포 횡단이다. 리서치 노트는
위키 항목 옆에 살지만 명시적인 읽기를 요구한다. 위키의 SessionStart 매처가 차이를
만든다.

## 8 · 루프별 산출물 트리

종료된 모든 루프는 `loops/NNN-<slug>/` 하위에 같은 트리를 남긴다. 노드를 호버하면
무엇인지, 어느 명령이 작성하는지, 어느 조항이 그것을 관장하는지가 보인다.

<ArtifactTree />

`results.tsv` 는 커밋 시리즈에서 재구성 가능하기 때문에 gitignore 된다 (조항 VIII —
"git이 곧 메모리"). 그 외 모든 것은 커밋된다. spec, clarifications, plan, 체크포인트,
report, 그리고 양방향 wiki-refs 다.

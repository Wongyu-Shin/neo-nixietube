# Feature Catalog — 29 entries

> Every harness feature this project has catalogued, classified by
> **axis1** (inner / outer) × **axis2** (pre-loop / in-loop / post-loop).
> Each entry's full spec lives in `harness/features/<slug>.md`; the
> research note in `harness/research/<slug>.md`; the rippable TC in
> `harness/tests/<slug>.sh`.

The catalog grew across **5 autoresearch loops** (initial 13 → Meta
HyperAgents +5 → google/agents-cli +3 → UX design +7 → docs +1 = 29).
Every iteration was guarded by `harness/composite-guard.sh` (Article VI),
so the count grew with **zero schema or cross-consistency regressions**.

---

## 1. The 2×3 axis matrix

Hover any cell to expand the features in that bucket. The matrix
distinguishes:

- **axis1: inner** = lives inside Claude Code (`.claude/`, settings, hooks,
  skills, slash commands).
- **axis1: outer** = lives outside (shell scripts, external orchestrators,
  CI matrices, MCP servers).
- **axis2: pre/in/post-loop** = which phase of the agent pipeline the
  feature gates.

<CatalogMatrix />

The colour intensity per cell encodes the **rippable_check robustness**
score (Article II). High-saturation cells have probes specific enough to
auto-trigger on a CC version bump (`statistical-tc-runner` + SessionStart
hook); pale cells need their probes hardened.

---

## 2. Cross-reference graph

Features rarely stand alone — most cite or are cited by others. The
composite guard (`crosscheck.sh` C6) enforces *symmetric* cross-references:
if feature A says "complements B", B's `Referenced by` section must list A.
The graph below is the resulting force-directed layout.

<CrossRefGraph />

Hubs (highest in-degree) are the features other features depend on most:

- **`harness-pause-resume`** — used as the mechanism by `harness-graduated-confirm`.
- **`statistical-tc-runner`** — the axis-1 meta-feature that makes every
  other `tc_script` load-bearing.
- **`gcli-agent-run-telemetry`** — substrate for plateau-detection,
  noise-aware-ratchet, and the eval-compare primitive.

Features at the periphery are intentionally orthogonal — keeping them
separate is what allows model/CC upgrades to absorb one without
collapsing the others (Article II).
